# Промт для /frontend — Ресёрч и план обновления UI/UX трекера задач

## Как использовать

Скопируй промт ниже и вставь его в новый диалог с Claude, вызвав скилл `/frontend`.

---

## Промт

```
<context>
Ты работаешь с существующим трекером задач — веб-приложением на следующем стеке:

- React 18 + TypeScript + Vite
- Tailwind CSS 4 + shadcn/ui (Radix primitives)
- Zustand (state management)
- Framer Motion (анимации)
- Supabase (бэкенд/БД)
- React Router DOM, recharts, react-day-picker, sonner, vaul, cmdk, lucide-react
- date-fns для дат

Текущая структура компонентов:
src/
├── components/
│   ├── AuthForm.tsx        — форма авторизации
│   ├── Layout.tsx          — главный лэйаут
│   ├── Navigation.tsx      — навигация (вкладки)
│   ├── TaskItem.tsx        — карточка задачи
│   ├── AddTaskForm.tsx     — форма добавления задачи
│   ├── DateNavigation.tsx  — переключение дат
│   ├── ProgressBar.tsx     — прогресс-бар выполнения
│   ├── ui/
│   │   ├── LoadingSpinner.tsx
│   │   └── DatePickerModal.tsx
│   └── views/
│       ├── TodayView.tsx       — задачи на сегодня
│       ├── TasksView.tsx       — общий список задач
│       ├── IdeasView.tsx       — идеи
│       ├── RecurringView.tsx   — регулярные задачи
│       └── AnalyticsView.tsx   — аналитика
├── hooks/ (useDatabase.ts, useAuth.tsx)
├── store/ (zustand store)
├── lib/ (utils, database, supabase)
└── types/ (index.ts)

Функциональность приложения:
- Авторизация по логину/паролю + демо-режим
- Вкладки: Сегодня, Задачи, Идеи, Регулярные, Аналитика
- Drag & drop для изменения порядка задач
- Регулярные задачи (ежедневно/по дням недели)
- Аналитика: среднее кол-во задач/день, статистика по дням, пропуски регулярных

Приложение рабочее, но визуально и по UX-опыту нуждается в обновлении.
</context>

<task>
Выполни полный ресёрч и подготовь детальный план обновления UI/UX трекера задач. Работа состоит из 3 этапов:

**ЭТАП 1: Аудит текущего состояния**
Прочитай КАЖДЫЙ компонент из src/components/ и src/components/views/. Для каждого зафиксируй:
- Текущую визуальную реализацию (стили, отступы, цвета, типографика)
- UX-проблемы (нет пустых состояний, нет скелетонов, плохая навигация, отсутствие обратной связи)
- Accessibility-проблемы (контрасты, aria-атрибуты, клавиатурная навигация)
- Адаптивность (работает ли на мобильных, планшетах)
- Консистентность (одинаковые ли отступы, шрифты, размеры во всех views)

**ЭТАП 2: Исследование лучших практик**
Исследуй и предложи обновления, опираясь на:
- Современные паттерны task management UI (Todoist, Things 3, TickTick, Linear)
- Принципы Material Design 3 / Apple HIG для визуальной системы
- Микровзаимодействия и анимации (у нас уже есть Framer Motion — используй его)
- Мобильная навигация (bottom tabs, swipe gestures)
- Систему пустых состояний, онбординга, обратной связи пользователю

**ЭТАП 3: Детальный план обновления**
Создай план в формате markdown-документа со следующей структурой:

1. **Дизайн-система** — цветовая палитра (light + dark mode), типографика, spacing scale, border-radius scale, shadow scale. Конкретные значения Tailwind-токенов.

2. **Компонент-по-компонент** — для каждого компонента:
   - Что сейчас
   - Что меняем и почему
   - Конкретные CSS/Tailwind изменения
   - Приоритет (P0 критично / P1 важно / P2 улучшение)

3. **Новые компоненты** — список компонентов, которых не хватает:
   - EmptyState (для каждого view)
   - Skeleton loaders
   - Toast notifications (sonner уже есть — как использовать лучше)
   - Onboarding flow
   - Command palette (cmdk уже есть — как интегрировать)

4. **UX-потоки** — улучшения пользовательских сценариев:
   - Добавление задачи (быстрое добавление, inline-редактирование)
   - Переключение между вкладками
   - Мобильный опыт (жесты, bottom sheet через vaul)
   - Клавиатурные сочетания

5. **Порядок выполнения** — разбивка на спринты:
   - Спринт 1: Дизайн-система + критичные фиксы
   - Спринт 2: Обновление основных views
   - Спринт 3: Новые компоненты и UX-потоки
   - Спринт 4: Полировка, анимации, тёмная тема

Каждый пункт плана ДОЛЖЕН содержать конкретные действия, а не общие рекомендации.
</task>

<constraints>
- ТОЛЬКО анализируй и планируй. НЕ пиши и НЕ изменяй код.
- Сохрани план в файл UI_REDESIGN_PLAN.md в корне проекта.
- Используй ТОЛЬКО те библиотеки, которые уже в package.json. НЕ предлагай новые зависимости.
- Все предложения ДОЛЖНЫ быть реализуемы с текущим стеком (React 18 + Tailwind 4 + shadcn/ui).
- Приводи конкретные примеры Tailwind-классов, а не абстрактные описания.
- Каждое визуальное изменение обосновывай с точки зрения UX (зачем пользователю это нужно).
- План на русском языке.
</constraints>

<output_format>
Один markdown-файл UI_REDESIGN_PLAN.md с чёткой структурой из 5 разделов (описаны выше).
Каждый раздел содержит таблицы, конкретные значения и приоритеты.
Объём: 800–1200 строк. Это рабочий документ, не презентация.
</output_format>
```
